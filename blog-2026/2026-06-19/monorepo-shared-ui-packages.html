Monorepo Shared UI Packages: One Codebase, Two Platforms

📅 2026-06-19
Learn how a single TypeScript package can power UI components for both React Native and Next.js, keeping styles and logic in sync while reducing duplication.
monorepo typescript react native

Insight

When you maintain separate repos for a mobile app and its web companion, UI inconsistencies and duplicated effort creep in fast. A monorepo with a shared ui package lets you write a component once, then render it with platform‑specific primitives (e.g., View vs div). By leveraging TypeScript path aliases and conditional exports, you keep the import surface identical across both apps, and the build tools automatically pick the right implementation.

Example

// packages/ui/src/Button.tsx
import { Platform, TouchableOpacity, Text, View } from 'react-native';

export const Button = ({ title, onPress }: { title: string; onPress: () => void }) =>
  Platform.OS === 'web' ? (
    <button onClick={onPress}>{title}</button>
  ) : (
    <TouchableOpacity onPress={onPress} style={{ padding: 12, backgroundColor: '#007AFF' }}>
      <Text style={{ color: '#fff' }}>{title}</Text>
    </TouchableOpacity>
  );

// tsconfig.json (root)
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": { "@ui/*": ["packages/ui/src/*"] }
  }
}

Now both apps/native/App.tsx and apps/web/pages/index.tsx can import @ui/Button without caring about the platform.

Takeaway

Structure your repo with a shared UI package, use TypeScript path aliases, and let conditional rendering handle platform differences. This cuts maintenance overhead and guarantees visual parity between mobile and web.
