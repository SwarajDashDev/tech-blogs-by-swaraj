Sharing UI Components Across React Native & Next.js with a Yarn Workspaces Monorepo

📅 2026-07-10
Learn how a Yarn Workspaces monorepo can let you write a single UI component library and reuse it in both React Native and Next.js, keeping type safety and build times tight.
monorepo typescript react-native

Insight

A single source of truth for UI components eliminates duplication and drift between your mobile and web codebases. By placing shared components in a packages/ui folder and using Yarn Workspaces, both the React Native app and the Next.js web app can import the same TypeScript files. The key is to keep platform‑specific implementations isolated (e.g., using *.native.tsx vs *.web.tsx) and let the bundler resolve the correct file.

Example

// packages/ui/Button.tsx (common API)
export const Button = ({ title, onPress }: { title: string; onPress: () => void }) => {
  // Platform‑specific render is delegated to the extensions below
  return <ButtonImpl title={title} onPress={onPress} />;
};

// packages/ui/Button.native.tsx
import { TouchableOpacity, Text } from 'react-native';
export const ButtonImpl = ({ title, onPress }) => (
  <TouchableOpacity onPress={onPress} style={{ padding: 12, backgroundColor: '#007aff' }}>
    <Text style={{ color: '#fff' }}>{title}</Text>
  </TouchableOpacity>
);

// packages/ui/Button.web.tsx
import { button } from './Button.module.css';
export const ButtonImpl = ({ title, onPress }) => (
  <button className={button} onClick={onPress}>{title}</button>
);

Takeaway

Keep shared UI in a dedicated package, use platform‑specific file extensions, and configure tsconfig.json paths ("@ui/*": ["../packages/ui/*"]). This approach gives you a single, type‑checked component library while letting each platform render the optimal native element.
